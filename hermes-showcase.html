Core Reframe

Hermes 不是“更像某个竞品”的 Agent，而是“更像系统”的 Agent。

如果只看模型强弱、UI 顺滑度、工具数量，你看到的只是表层。Hermes 真正值得讨论的，是它把 Agent 做成一种像协议一样有状态、像账户一样有历史、像操作系统一样能长期运行的基础设施。

Interpretation Layer

blocmates 的价值，不是夸 Hermes，而是给了它一个可理解框架。

最聪明的解读不是说 Hermes “很强”，而是把它解释成一个 persistent、self-hosted、越用越聪明的系统，并用一个很适合社区理解的框架组织起来：Knowledge Layer、Execution Layer、Output Layer。

Learning Loop

Hermes 所说的“自我积累 / 自我学习”，本质上是系统层的持续沉淀，而不是模型自己训练自己。

它真正做的不是改模型权重，而是把一次任务里形成的事实、偏好、历史和方法，分别写入不同的持久层，再在后续会话中重新调出来使用。

Learning Loop Diagram

01

Memory

稳定事实、偏好和约定写入 `MEMORY.md` / `USER.md`。

02

Session Recall

完整历史进入 `state.db`，靠 FTS5 和 `session_search` 被重新唤回。

03

Skills

复杂 workflow 沉淀成 skill，变成 procedural memory。

04

Compression

长会话前先 flush，再 structured summary，减少经验蒸发。

结果是：任务里的事实、方法、历史与偏好，不会只停留在一次回答里，而会在下一次会话重新进入系统。

Knowledge Layer

真正厉害的不是“记住”，而是“有边界地记住”。

Hermes 关心的不是“记得更多”，而是“记得更对”。长期记忆被设计成 bounded, curated memory，而不是无上限上下文堆积。

Chart 3

内建记忆组件

组件 作用 含义
MEMORY.md 记录环境事实、约定、经验 保留高价值、低变动信息
state.db 历史会话库，支持检索与摘要 历史不是常驻 prompt，而是按需唤回
Snapshot Design

记忆是快照，不是流

记忆在会话开始时注入，中途不会实时刷新；新的记忆先落盘，再在下一次会话中进入上下文。

  • 区分常驻、检索、技能沉淀
  • 更接近人的认知结构
  • 核心仍是保有目标与边界定义权

Memory Options

内建 memory 和 Honcho，不是替代关系，而是层级关系。

Hermes 的内建记忆强调稳定、简洁和 prompt 友好；Honcho 则是在其之上增加更深的用户建模与语义搜索。

Built-in

内建 memory

稳定、克制、适合长期缓存，适合大多数个人用户与本地部署。

Honcho

Provider Plugin

更偏“推断你”的外接后端，适合更深用户建模、多代理隔离与语义检索。

Takeaway
Hermes 真正像的，不是无边界的第二大脑，而是一个经过工程约束的数字自我延伸器官。

Skills System

如果记忆解决“别忘”，那技能解决的是“别白做”。

Hermes 最像“会成长的系统”的地方，不只是 memory，而是 skills。记忆保存事实和偏好，技能则把经验压缩成可复用的方法。

Why It Matters

经验开始具备可编译性

Skills 不是一句 prompt 模板，而是 on-demand knowledge documents。Hermes 通过 progressive disclosure 按需读取技能，不一次性把所有东西塞进上下文。

  • 节省 token
  • 降低上下文噪声
  • 让一次次任务有机会沉淀成结构化方法
Interpretation
技能系统真正可贵的，不是“下次回答更像一点”，而是把手艺从一次性的会话热闹，压缩成可调用、可复用、可改进的知识资产。

Execution Layer

Agent 的意义，不是会调工具，而是会拆工作。

Hermes 真正强的执行层，不只是 persistent machine access，而是 delegation：生成 isolated child agents 来并行处理任务，每个 subagent 拥有独立 conversation、terminal session 与 toolset。

Chart 6

适合委派 vs 不适合委派

适合 原因 不适合
研究型子问题 上下文快速膨胀，适合独立处理 单步工具调用
平行独立工作流 可显著缩短总完成时间 非常机械的短流程
需要 fresh context 的任务 避免主线程被噪音污染 需要频繁向用户澄清的任务
推理密集型子任务 让主代理保留更高层统筹 小规模文件微调
Delegation

上下文隔离

返回给主代理的是总结结果，而不是全部中间过程，这让并行不是多线程，而是真正的任务隔离。

Cron

调度进入执行面

自然语言、cron expression、interval、ISO 时间戳都能成为触发条件，到期任务由 gateway daemon 驱动。

隔离子代理

The delegate_task tool spawns child AIAgent instances with isolated context,

Each child gets a fresh conversation and works independently —

cron/scheduler.py

调度与后台执行

Provides tick() which checks for due jobs and runs them.
The gateway calls this every 60 seconds from a background thread.

Uses a file-based lock (~/.hermes/cron/.tick.lock)
so only one tick runs at a time.

Output Layer

Agent 真正落地，不是回答得好，而是结果能回到你的生活里。

很多 Agent 死在最后一步：结果困在对话框里。Hermes 把“输出回投”当成系统能力，所以最该先配的往往不是更花哨的 prompt，而是 gateway。

Gateway First

别只盯着 CLI

  • 离开电脑后仍能触达 agent
  • 结果能自动回投到习惯通道
  • 工作流不再依赖单一终端窗口
Takeaway
一旦结果能稳定流回你已经习惯的消息系统，Agent 才真正进入你的日常，而不是停留在 demo 阶段。

Practice Guide

别把 Hermes 当玩具，要把它当账户体系来养。

如果真想用起来，而不只是截图发朋友圈，最好的方法不是上来追求最花哨的玩法，而是先把它养成一个稳定工作的系统账户。

01
第一步

先跑通本体

安装 Hermes，跑通 hermes model、gateway、hermes，先确保本体稳定工作。

02
第二步

先理解迁移，再决定导入

用 claw migrate --dry-run 观察旧资产如何映射，不要把“迁移成功”简化成“文件复制成功”。

03
第三步

选择固定高频任务

比如投研摘要、周报整理、会议纪要归档，让 Hermes 开始积累真实方法，而不是空转。

04
第四步

配一个消息出口和图形界面

让它脱离单一终端，进入你的日常工作流。Gateway + Web UI 往往比继续微调 prompt 更重要。

05
第五步

最后再追求高阶玩法

Honcho、远程部署、多代理分工都很值得玩，但应建立在稳定回路先跑起来的前提上。
